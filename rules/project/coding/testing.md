# 测试验证方法

## 验证策略

*   修改代码后必须运行相关测试
*   提交代码前必须通过 lint 和 typecheck
*   优先修复代码而非修改测试用例
*   静态检查问题与测试失败同等严重，不得以任何理由跳过或降级处理

## 测试层级定义

| 层级 | 覆盖范围 | 验证目标 | 通过标准 |
|------|----------|----------|----------|
| **单元测试** | 单个函数/方法 | 逻辑正确性 | 100% 覆盖分支 |
| **模块测试** | 组件/工具类 | 接口正确性 | 所有公开接口验证通过 |
| **集成测试** | 多模块协作流程 | 端到端正确性 | 完整流程无错误 |
| **关键逻辑覆盖测试** | 核心业务逻辑 | 核心逻辑正确性 | 所有关键逻辑分支验证通过 |
| **全流程黄金测试** | 完整端到端执行流程 | 一致性保障 | 输出与基准完全一致 |

## 测试完整性要求

技能开发完成后必须通过以下测试类型，任一项不通过即禁止合入或发布：

| 测试类型 | 说明 |
|----------|------|
| **单元测试** | 函数级逻辑验证 |
| **模块测试** | 组件级接口验证 |
| **集成测试** | 多模块协作流程验证 |
| **关键逻辑覆盖测试** | 核心业务逻辑验证 |
| **全流程黄金测试** | 端到端基准比对 |

## 脚本可测试性设计

检查脚本设计是否支持：

*   **单元测试**：函数独立、依赖注入
*   **可观测性**：日志、指标
*   **可调试性**：断点、错误信息

## 断言保护

`security_level ≥ high` 时必须声明：

*   **pre_conditions**：执行前断言
*   **post_conditions**：执行后断言

## Fallback 策略

涉及不可逆操作时必须声明 `fallbacks` 错误恢复策略。
