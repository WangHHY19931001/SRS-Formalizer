# BDD（行为驱动开发）

> BDD 是一种软件开发方法论，通过描述系统的预期行为来驱动开发，强调团队沟通和协作。

## 核心流程

### 阶段一：发现

团队成员共同探索需求，理解业务场景：

1. **用户角色**：定义谁会使用系统
2. **用户目标**：用户想要达成什么
3. **行为场景**：描述系统应该如何响应

### 阶段二：描述

使用自然语言描述系统行为：

```gherkin
Feature: 购物车结算
  As a 顾客
  I want to 结算购物车中的商品
  So that I can 完成购买

  Scenario: 购物车中有商品
    Given 购物车中有 2 件商品
    When 点击"结算"按钮
    Then 显示订单确认页面
    And 订单金额正确计算

  Scenario: 购物车为空
    Given 购物车为空
    When 点击"结算"按钮
    Then 显示提示"购物车为空"
```

### 阶段三：验证

将自然语言描述转化为自动化测试：

1. 为每个场景编写测试步骤
2. 实现测试步骤的自动化绑定
3. 运行测试验证行为是否符合预期

### 阶段四：实现

根据测试驱动开发实现：

1. 运行测试确认失败（红）
2. 编写代码使测试通过（绿）
3. 重构代码优化实现（重构）

## 关键实践

*   **自然语言**：使用团队共同理解的语言描述行为
*   **场景覆盖**：覆盖正常、异常、边界等多种场景
*   **实时反馈**：测试结果即时反馈，快速迭代
*   **文档即代码**：Gherkin 描述既是文档也是测试

## 与 SOP 的协同

BDD 的"行为描述"与 SOP 的"拷问循环"协同：

1. 拷问阶段澄清业务场景和行为预期
2. BDD 将澄清后的需求转化为可执行的行为描述
3. 行为描述作为开发和测试的共同依据
