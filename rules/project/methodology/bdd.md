# BDD（行为驱动开发）

> BDD 是一种软件开发方法论，通过描述系统的预期行为来驱动开发，强调团队沟通和协作。

## 核心流程

### 阶段一：发现

团队成员共同探索需求，理解业务场景：

1. **用户角色**：定义谁会使用系统
2. **用户目标**：用户想要达成什么
3. **行为场景**：描述系统应该如何响应

### 阶段二：描述

使用 Gherkin 语言描述系统行为。Gherkin 关键词分三类：

**主要关键词（构建文件骨架）**

| 关键词 | 别名 | 用途 |
|:---|:---|:---|
| `Feature` | — | 文件顶级关键词，每文件唯一，后跟业务意图说明 |
| `Rule` | — | 可选（Gherkin v6+），在 Feature 内按业务规则分组场景 |
| `Example` | `Scenario` | 具体测试用例，描述一个业务规则下的一个具体例子 |
| `Background` | — | 每个场景运行前自动执行的公共前置步骤，每文件最多一个 |
| `Scenario Outline` | `Scenario Template` | 场景大纲，配合 `Examples` 表格以多组数据驱动同一场景 |
| `Examples` | `Scenarios` | 紧跟 `Scenario Outline`，以表格形式提供多组测试数据 |

**步骤关键词（定义操作步骤）**

| 关键词 | 用途 |
|:---|:---|
| `Given` | 前置条件：场景开始前的系统状态，须完整枚举所有状态变量 |
| `When` | 触发事件：用户或外部系统的具体动作，每场景应只有一个 |
| `Then` | 预期结果：必须细化到具体字段、状态码或界面反馈，禁止模糊表述 |
| `And` | 正向连接词，连接多个同类型步骤，语义继承所连接步骤的类型 |
| `But` | 否定/对比连接词，`And` 的否定形式，语义继承所连接步骤的类型 |
| `*`（星号） | 通配符，可替代任意步骤关键词，适合列举性断言 |

**次要关键词（提供额外数据或元信息）**

| 关键词 | 用途 |
|:---|:---|
| `"""`（文档字符串） | 传递多行文本作为步骤参数（JSON、XML、长文本等） |
| `\|`（数据表） | 在单个步骤后传递结构化表格数据，不驱动场景重复执行 |
| `@`（标签） | 给 Feature/Rule/Scenario 打标记，用于筛选和分组执行 |
| `#`（注释） | 添加注释，`#` 须为行首第一个非空白字符 |

**本地化**：使用中文时在文件开头声明 `# language: zh-CN`，可用"功能/场景/假如/当/那么/而且/但是"等中文关键词。

```gherkin
@checkout
Feature: 购物车结算
  作为注册用户，我希望结算购物车中的商品，以便完成购买。

  Background:
    Given 用户已登录且账户状态正常

  Rule: 购物车非空时才允许结算

    Scenario: 购物车中有商品
      Given 购物车中有以下商品
        | 商品名称 | 数量 | 单价（元） |
        | 无线耳机 | 1    | 299        |
      When 用户点击"立即结算"
      Then 系统返回 200 状态码
      And 订单总金额为 299 元

    Scenario: 购物车为空
      Given 购物车中没有任何商品
      When 用户点击"立即结算"
      Then 系统返回 400 状态码
      And 响应体包含错误码 "CART_EMPTY"

  Rule: 多种支付方式均可完成结算

    @data-driven
    Scenario Outline: 不同支付方式结算
      Given 购物车中有 1 件价值 <金额> 元的商品
      When 用户选择 <支付方式> 完成支付
      Then 订单状态为 "<订单状态>"

      Examples:
        | 支付方式 | 金额 | 订单状态 |
        | 支付宝   | 100  | PAID     |
        | 微信支付 | 200  | PAID     |
```

### 阶段三：验证

将自然语言描述转化为自动化测试：

1. 为每个场景编写测试步骤
2. 实现测试步骤的自动化绑定
3. 运行测试验证行为是否符合预期

### 阶段四：实现

根据测试驱动开发实现：

1. 运行测试确认失败（红）
2. 编写代码使测试通过（绿）
3. 重构代码优化实现（重构）

## 关键实践

*   **自然语言**：使用团队共同理解的语言描述行为
*   **场景覆盖**：覆盖正常、异常、边界等多种场景
*   **实时反馈**：测试结果即时反馈，快速迭代
*   **文档即代码**：Gherkin 描述既是文档也是测试

## 与 SOP 的协同

BDD 的"行为描述"与 SOP 的"拷问循环"协同：

1. 拷问阶段澄清业务场景和行为预期
2. BDD 将澄清后的需求转化为可执行的行为描述
3. 行为描述作为开发和测试的共同依据
