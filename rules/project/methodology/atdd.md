# ATDD（验收测试驱动开发）

> ATDD 是一种敏捷开发方法，强调在编写代码之前先编写验收测试，确保代码符合业务需求。

## 核心流程

### 阶段一：协作讨论

与业务人员、测试人员、开发人员共同讨论需求，明确验收标准：

1. **用户故事**：描述用户想要什么，以及为什么需要
2. **验收标准**：明确"完成"的定义，用可测试的语言描述
3. **场景**：列举不同情况下的行为预期

### 阶段二：编写验收测试

基于验收标准编写自动化测试用例：

```gherkin
Feature: 用户登录
  Scenario: 正确的用户名和密码
    Given 用户在登录页面
    When 输入正确的用户名和密码
    Then 成功登录并跳转到首页

  Scenario: 错误的密码
    Given 用户在登录页面
    When 输入正确的用户名和错误的密码
    Then 显示错误提示"密码错误"
```

### 阶段三：开发实现

编写代码使验收测试通过：

1. 实现最小化功能以通过测试
2. 确保所有测试用例都能通过
3. 不添加超出测试范围的功能

### 阶段四：持续集成

将代码集成到 CI/CD 流程中：

1. 每次提交自动运行所有验收测试
2. 测试失败时阻止部署
3. 定期回顾测试覆盖率

## 关键实践

*   **三方协作**：业务、测试、开发共同参与
*   **测试先行**：先写测试，再写实现
*   **持续反馈**：测试结果及时反馈给团队
*   **测试维护**：测试与代码同步更新

## 与 SOP 的协同

ATDD 的"验收测试先行"与 SOP 的"拷问循环"互补：

1. 拷问阶段产出的 `PLAN.md` 包含验收标准
2. ATDD 将验收标准转化为自动化测试
3. 测试通过作为代码合入的 gate condition
