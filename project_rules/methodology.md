# 工作方法论

> 核心公式：决策树穷尽 + 事实交叉验证 + 渐进式文档沉淀。不限于 AI 编程，可迁移至任何复杂决策场景。

## 四层架构

### 第一层：认知引擎层 —— 苏格拉底式"决策树拷问"

**规则一（分支穷尽）**：将模糊目标视为一棵"决策树"。AI 不生成枝叶，而是沿着每一个 if-else 分支追问，直到覆盖所有潜在变量和极端情况。

**规则二（单线程聚焦）**：每次只抛出一个问题，并附带 AI 自身的"推荐答案"。避免信息过载，让人类通过"同意 / 修改 / 否决"快速推进，保持认知负担最小化。

**规则三（信息自足）**：遵循"能查就不问"原则。AI 必须先扫描现有上下文（代码库、历史文件），只有无法从客观事实中推导出的主观决策点（如"红色还是蓝色"），才抛给人类。

### 第二层：验证校准层 —— 领域驱动的事实交叉验证

**术语一致性校验**：捕捉描述中的模糊词汇，与项目既有的术语表 / 数据字典进行碰撞。发现冲突（如用户说的"部分退款"与代码中的"整单退款"矛盾）时，强制中断拷问并抛出矛盾点。

**代码事实核对**：不轻信口头描述，直接读取实际代码逻辑来验证假设的真伪。

### 第三层：产出沉淀层 —— 渐进式"活文档"构建

拷问不是单纯的口水仗，而是边聊边写。

**内联更新**：每当敲定一个术语或决策，立即将其固化到特定的结构化文档中（如 `CONTEXT.md` 存储术语，`ADR/` 存储架构决策记录）。

**最终缝合**：拷问结束时，AI 不输出代码，而是输出一份无歧义、无缺失、带优先级的静态计划书（`PLAN.md`）。这份计划书是后续"执行 AI"的唯一输入源。

### 第四层：流程控制层 —— 范围与保真度管理

**切分大块（Token 预算意识）**：若目标过大（超 12 万 token 上下文），强制拆解为多个子目标分别拷问，防止模型在长上下文中"变笨"或丢失细节。

**区分问题保真度（Fidelity Filter）**：建立过滤机制。低保真度问题（如 API 路由、数据结构）直接由拷问解决；高保真度问题（如 UI 交互手感、视觉审美）禁止拷问，必须先快速原型验证，拿到感官反馈后再回到拷问流程。

## 标准化执行工作流（SOP）

| 阶段 | 步骤 | 关键动作 |
|------|------|----------|
| Phase 1 | 输入初始想法 | 向 AI 丢入一个粗糙的、甚至只有一两句话的意图。 |
| Phase 2 | 开启"拷问循环" | AI 读取现有文档 / 代码 → 提出 1 个精准的决策性问题（附推荐答案）→ 人类作答 → AI 修正内部认知模型 → 重复此循环 N 次（直到 AI 评估决策树已达 100% 覆盖）。 |
| Phase 3 | 事实校对（增强版） | 若启用 `with-docs`，AI 将在拷问间隙主动检索代码，即时反击错误假设。 |
| Phase 4 | 生成"冻结"规格书 | 终止拷问。AI 将所有达成共识的结论整理为 `PLAN.md`（含技术方案、文件改动清单、测试策略）。 |
| Phase 5 | 上下文隔离执行 | 清空当前对话上下文（关键！），开启新会话，仅将这份 `PLAN.md` 喂给编程 AI，令其严格执行。防止拷问阶段的历史"噪声"污染最终代码生成。 |
